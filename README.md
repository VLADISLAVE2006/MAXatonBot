# API для бота MAX (регистрация пользователей)

Этот API обрабатывает регистрацию пользователей: сохранение согласия на обработку ПД и запись ФИО.

## Базовый URL

- Для локальной разработки: `http://localhost:8080`
- Когда поднимете на сервере — замените на реальный адрес (например, `https://api.ваш-домен.ru`)

## Аутентификация

**Все запросы (кроме `/health`) должны содержать заголовок:** X-API-Key: very-secret-key-2025


> ⚠️ Этот ключ совпадает с тем, что указан в `.env` на сервере. Убедись, что в вашем боте он тоже задан (например, в переменной окружения `API_KEY`).

## Проверка работоспособности (опционально)

**GET** `/health` — ответ `200 OK` с текстом `"OK"`.  
Этот эндпоинт публичный, помогает убедиться, что сервер жив.

## Эндпоинты

### 1. Сохранение согласия

**POST** `/api/user/consent`

**Заголовки:**

Content-Type: application/json

X-API-Key: very-secret-key-2025


**Тело (JSON):**
```json
{
  "user_id": 123456789,
  "agreed": true,
  "agreement_version": "1.0"
}
```

## 2. Сохранение ФИО

**POST** `/api/user/profile`

**Заголовки:** те же (`Content-Type` и `X-API-Key`)

**Тело (JSON):**

```json
{
  "user_id": 123456789,
  "full_name": "Иванов Иван Иванович"
}
```

## 3. Назначение организатора (только для админа)

**POST** `/api/admin/set-role`

**Заголовки:** `Content-Type: application/json`, `X-API-Key: ...`

**Тело (JSON):**
```json
{
  "admin_user_id": 987654321,
  "target_user_id": 123456789,
  "role": "organizer"
}
```



## 5. Что ещё нужно учесть?

- При первом запуске API администратор создаётся автоматически. Если позже вы захотите сменить администратора, измените `ADMIN_USER_ID` в `.env` и перезапустите сервер — роль старого админа не откатится автоматически (останется `admin`). Чтобы этого избежать, можно донастроить логику, но для хакатона это излишне.
- Роль `admin` даёт доступ к эндпоинту `/api/admin/set-role`. Других административных функций пока нет.
